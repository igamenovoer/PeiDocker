## Summary

- The change is directionally strong: it matches the repository's existing preference for manual-trigger heavy Docker verification while moving orchestration into pytest for better selection and reporting.
- The SSH-first approach aligns well with PeiDocker's real login-path behavior, especially for `conda-environment` and other examples that rely on shell initialization.
- The most important missing design detail is runtime host-port isolation. The examples hardcode SSH and service ports, and the current design only isolates image tags and Compose project names.
- The proxy example contract is still too loose. The artifacts say cleanup should be "observable" but do not define which runtime signals prove the documented proxy and APT cleanup behavior.
- Mixed `${...}` and `{{...}}` examples introduce a subtle harness design constraint that should be decided explicitly before implementation starts.

## Findings

### Must Fix / Blocking

1. **The design isolates image tags and Compose project names, but it does not isolate the examples' fixed host ports, so the suite can still fail for port-collision reasons unrelated to the scenario under test.**

The current design calls out run-scoped image tags and Compose project names in `design.md:55-58`, and the tasks carry that through in `tasks.md:5-6,12,18-21`. That is not sufficient for this repo's example set. The packaged examples hardcode SSH and service host ports such as `2222` through `2234` and `8080` / `3000-3002` in `src/pei_docker/examples/basic/minimal-ssh/user_config.yml:5-11`, `src/pei_docker/examples/basic/gpu-container/user_config.yml:5-21`, `src/pei_docker/examples/basic/host-mount/user_config.yml:21-23`, `src/pei_docker/examples/basic/port-mapping/user_config.yml:5-19`, and `src/pei_docker/examples/basic/env-passthrough/user_config.yml:17-23`. The spec also requires host-side port assertions in `spec.md:44-46`, and the existing heavy shell suite avoids this class of flake by binding SSH to an ephemeral port with `docker run -p 0:22` in `tests/functional/entrypoint-non-tty-default-blocking/run.bash:246-265`. This change should explicitly require per-run free-port allocation and config rewriting for SSH and declared service ports, with the resolved ports surfaced back into the assertion helpers.

### Should Fix / Important

2. **The proxy scenario still lacks a concrete observable contract for what "cleanup behavior" means at runtime, which makes the most environment-sensitive example easy to under-test.**

The design correctly requires env-derived proxy injection and loopback normalization in `design.md:69-73`, and the spec requires the example to use invoking proxy settings and reflect runtime cleanup behavior in `spec.md:48-50,98-107`. The problem is that the current wording in `design.md:83` only says "runtime proxy cleanup behavior is observable" without defining the observables. In the existing docs, global proxy cleanup and APT proxy cleanup are separate behaviors: `docs/manual/guides/proxy-configuration.md:21-23,31-39,47-48` and `docs/examples/basic/07-proxy-setup.md:37-41` distinguish shell proxy variables from APT-specific proxy config, while the example itself sets `remove_after_build: true` and `keep_proxy_after_build: false` in `src/pei_docker/examples/basic/proxy-setup/user_config.yml:12-21`. The change should say exactly what the harness must check, for example whether a non-root login shell lacks `http_proxy` / `https_proxy`, whether the APT proxy file is absent, and which missing host prerequisites cause a skip versus a hard failure.

3. **Run-scoped image-name rewriting is underspecified for mixed `${...}` and `{{...}}` examples, and the current plan risks validating a harness-mutated config instead of the example's real compose-time behavior.**

The design says each run will "rewrite output image names to unique, run-scoped tags before build" in `design.md:55-57`. That is straightforward for fixed image names, but `env-passthrough` deliberately uses mixed resolution semantics: `src/pei_docker/examples/basic/env-passthrough/user_config.yml:17-23` sets `output: "${PROJECT_NAME:-pei-env-pass}:{{TAG:-dev}}"` and uses `{{WEB_HOST_PORT:-18080}}` for runtime passthrough. The repository docs make the distinction explicit in `docs/manual/concepts/environment-variables.md:16-20,34-42`: `${...}` is resolved during `configure`, while `{{...}}` must remain for Docker Compose to resolve later. Without an explicit rule here, implementation could erase the passthrough contract by post-processing the generated compose or flattening the mixed syntax too early. The artifacts should state that uniqueness for mixed-mode examples comes from controlled configure-time env such as `PROJECT_NAME`, while passthrough placeholders like `TAG` remain part of the scenario contract.

### Q1) Should explicit invocation be exposed as both a dedicated Pixi task and a raw pytest command?

- Decision: `Deferrable`

#### Why this matters

- The repo already exposes heavy functional verification through more than one stable entrypoint.
- Picking only one path is possible, but choosing both improves discoverability without changing default fast-test behavior.

#### Examples

- Today developers can run the heavy entrypoint suite through a direct shell path, a wrapper script, or `pixi run test-entrypoint-e2e`.
- A new pytest lane will be easier to adopt if the docs show a raw pytest command for targeted reruns and a Pixi task for the standard "known good" entrypoint.

#### Why this question arises in code

- `pyproject.toml:143-147` currently exposes `test` and `test-entrypoint-e2e`.
- `docs/developer/testing.md:9-14,37-43` and `tests/functional/entrypoint-non-tty-default-blocking/README.md:48-58` show the established pattern of documenting both a stable wrapper and a Pixi task.

#### Options

**Option A: Document only a raw pytest command.**

- Code impact:
  - Add the pytest marker/path and docs, but no new Pixi task.
- User-facing difference:
  - Developers need to remember a longer pytest invocation for the heavy lane.
- Pros:
  - Smaller surface area.
  - Avoids adding another task name.
- Cons:
  - Less consistent with the existing heavy-suite ergonomics.
  - Makes the "official" invocation path less obvious in docs and CI snippets.

**Option B: Expose both a dedicated Pixi task and a raw pytest command.**

- Code impact:
  - Add marker/path-based pytest invocation plus one thin Pixi task that delegates to it.
- User-facing difference:
  - Developers get one short standard command and one flexible direct pytest form.
- Pros:
  - Matches the current repo pattern for heavy Docker verification.
  - Keeps targeted local debugging easy without forcing everyone through Pixi syntax.
- Cons:
  - Slightly more documentation and task maintenance.

> **PROPOSED: Expose both a Pixi task and a raw pytest command.**
> Rationale: The repository already treats heavy functional verification as a first-class but opt-in workflow, and it exposes that workflow through both docs and a dedicated task in `pyproject.toml:147` plus `docs/developer/testing.md:9-14,37-43`. Mirroring that pattern keeps the new suite native to the repo, while raw pytest still provides the best developer experience for targeted reruns and single-scenario debugging.

### Q2) For the GPU example, should hardware-specific checks be capability-gated assertions or a hard requirement for any selected run?

- Decision: `Blocking`

#### Why this matters

- This choice determines whether the suite can run partially on ordinary developer machines or whether selecting the GPU scenario implies an NVIDIA-capable host.
- The answer changes both skip/fail semantics and the shape of the assertion helpers.

#### Examples

- A non-GPU host can still verify that the generated Compose/runtime config includes the expected GPU-related settings.
- Only a GPU-capable host can verify device visibility or run a CUDA-level smoke command.

#### Why this question arises in code

- The requirement in `spec.md:28-30` says the non-root user observes the expected GPU-related runtime configuration.
- The example sets `device.type: gpu` in both stages and exports `NVIDIA_VISIBLE_DEVICES` and `NVIDIA_DRIVER_CAPABILITIES` in `src/pei_docker/examples/basic/gpu-container/user_config.yml:12-21` and `docs/examples/basic/02-gpu-container.md:19-36`.
- The design already hints at gating in `design.md:78-79,103`.

#### Options

**Option A: Capability-gated hardware checks.**

- Code impact:
  - Always assert the config-level GPU contract.
  - Add an extra host-capability probe before hardware-dependent commands.

```markdown
if host_has_gpu_runtime():
    assert_gpu_visible_in_container()
else:
    pytest.skip("GPU runtime not available on host")
```

- User-facing difference:
  - Non-GPU hosts can still run the rest of the suite and get a clear skip for hardware verification.
- Pros:
  - Fits local development reality.
  - Separates "config was generated correctly" from "host has GPU runtime support".
- Cons:
  - Slightly more branching in the harness.

**Option B: Require an NVIDIA-capable host whenever the GPU scenario is selected.**

- Code impact:
  - Fail fast if the host cannot satisfy the scenario.
- User-facing difference:
  - The scenario is unusable on common non-GPU machines.
- Pros:
  - Stronger end-to-end guarantee when it runs.
- Cons:
  - Much narrower usability.
  - Pushes an implementation-specific host requirement into a basic-example verification suite.

> **PROPOSED: Use capability-gated hardware checks, while always asserting the generated GPU-related runtime configuration.**
> Rationale: The example's documented contract is partly config-level (`device.type: gpu`, `NVIDIA_VISIBLE_DEVICES`, `NVIDIA_DRIVER_CAPABILITIES`) and only partly hardware-level, as shown in `src/pei_docker/examples/basic/gpu-container/user_config.yml:12-21` and `docs/examples/basic/02-gpu-container.md:32-36`. That makes a two-layer assertion model the best fit for the repo: ordinary developers can still validate the example wiring, and GPU-capable hosts can add the stronger device-visibility check without redefining the whole suite as GPU-only.
>

### Q3) How should run-scoped uniqueness preserve the compose-time contract of mixed `${...}` and `{{...}}` examples?

- Decision: `Blocking`

#### Why this matters

- The helper strategy chosen here affects build naming, cleanup, and whether the suite still validates real PeiDocker substitution semantics.

#### Examples

- `env-variables` expects concrete `${...}` resolution before runtime.
- `env-passthrough` expects `{{...}}` values to survive `configure` and resolve only when Compose starts the container.

#### Why this question arises in code

- `docs/manual/concepts/environment-variables.md:5-20,34-42` explicitly defines a two-stage substitution model.
- `src/pei_docker/examples/basic/env-passthrough/user_config.yml:17-23` mixes both forms in the image output and runtime environment fields.

```markdown
output: "${PROJECT_NAME:-pei-env-pass}:{{TAG:-dev}}"
```

#### Options

**Option A: Rewrite the generated compose or flattened image names after `configure`.**

- Code impact:
  - Patch generated artifacts directly to force unique image names and ports.
- User-facing difference:
  - The harness may no longer be validating the example as a normal user would run it.
- Pros:
  - Straightforward to implement for uniqueness.
- Cons:
  - Risks erasing the compose-time behavior the example is supposed to demonstrate.
  - Makes cleanup logic depend on post-generation mutations.

**Option B: Drive uniqueness through controlled env inputs while preserving passthrough placeholders that are part of the scenario.**

- Code impact:
  - Set run-scoped configure-time env such as `PROJECT_NAME` and scenario-scoped Compose env such as `TAG` and `WEB_HOST_PORT`, then let normal `configure` and `docker compose up` semantics do the rest.
- User-facing difference:
  - The test exercises the same substitution model users are documented to rely on.
- Pros:
  - Aligns with the repository's env-substitution architecture.
  - Keeps assertion logic closer to real user workflows.
- Cons:
  - Requires more careful helper APIs for per-scenario env injection.

> **PROPOSED: Preserve the documented `${...}` versus `{{...}}` split and drive uniqueness through controlled env inputs instead of post-generation rewrites for mixed-mode examples.**
> Rationale: The repository explicitly documents two different substitution moments in `docs/manual/concepts/environment-variables.md:5-20,34-42`, and `env-passthrough` is one of the examples meant to prove that distinction. Using run-scoped configure-time env such as `PROJECT_NAME` plus scenario-scoped Compose env such as `TAG` and `WEB_HOST_PORT` keeps the harness aligned with normal PeiDocker behavior and avoids validating a harness-specific config mutation.
>
